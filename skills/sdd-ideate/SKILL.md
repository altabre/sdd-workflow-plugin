# SDD Ideate — 想法固化（阶段 1.5）

## 职责

通过结构化对话，将模糊的想法/需求固化为一份清晰的 PRD 文档，作为后续 `sdd-prd-to-spec` 的输入。

## 触发条件

- 用户没有 PRD 文档
- 需求还没有固化，只有一个模糊的想法
- `config.yaml` 中 `inputs.prd: optional`

## 执行步骤

### 1. 读取项目上下文

先读取 `.sdd/onboard.md` 了解项目背景，再读取需求目录路径（由 sdd 主编排传入）。

### 2. 理解初始想法

请用户用 1-3 句话描述这个想法/需求是什么。不要展开，只是先理解大方向。

### 3. 逐步澄清（每次只问一个问题）

每条消息只问一个问题，等用户回答后再问下一个。问题围绕：

- **目标用户** — 谁会用这个功能？
- **核心问题** — 这个功能解决了什么痛点？
- **成功标准** — 怎么判断这个功能做成了？
- **边界** — 明确不做什么（YAGNI）
- **约束** — 技术约束、时间约束、兼容性要求

遇到范围过大的需求（包含多个独立子系统），先帮用户拆分，确认第一个子需求再继续。

### 4. 提出 2-3 种方案

基于澄清结果，提出 2-3 种实现思路，每种包含：
- 核心思路
- 优点
- 缺点/风险

给出推荐方案和理由，等用户选择。

### 5. 呈现设计并获得确认

分节展示设计，每节结束后问用户是否认可，不认可则修改：

1. **功能范围** — 包含什么，不包含什么
2. **用户流程** — 主流程 + 异常流程
3. **数据模型** — 涉及的核心数据结构（如有）
4. **接口定义** — API 端点或组件接口（如有）
5. **验收标准** — 可量化的完成判断依据

### 6. 写入 PRD 文档

设计确认后，将结果写入 `.sdd/{需求目录}/inputs/prd.md`：

```markdown
# PRD: {需求名称}

## 背景与目标
{问题描述，目标用户，核心痛点}

## 功能范围
### 包含
- ...

### 不包含（明确排除）
- ...

## 用户流程
### 主流程
1. ...

### 异常流程
- ...

## 数据模型
{核心数据结构，如无则省略}

## 接口定义
{API 或组件接口，如无则省略}

## 验收标准
- [ ] ...
- [ ] ...

## 约束与风险
- ...
```

### 7. 更新状态

将 state.yaml 中 `current_stage` 更新为 `PRD_TO_SPEC`，并写入历史记录：

```yaml
history:
  - stage: IDEATE
    status: completed
    timestamp: "{当前时间}"
```

## 完成条件

- [x] `.sdd/{需求目录}/inputs/prd.md` 已写入
- [x] state.yaml 已更新为 PRD_TO_SPEC 阶段

## 产出物

- `.sdd/{需求目录}/inputs/prd.md`
