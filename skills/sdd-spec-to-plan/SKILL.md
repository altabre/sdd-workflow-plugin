# SDD Spec to E2E Plan — 规格转 E2E 方案（阶段 4.1）

## 职责

基于技术规格，生成完整的端到端实施方案，指导后续开发。

## 路径约定
本阶段所有文件读写基于当前需求目录 `.sdd/{需求目录}/`：
- spec → `.sdd/{需求目录}/docs/spec.md`
- plan → `.sdd/{需求目录}/docs/plan.md`
- state → `.sdd/{需求目录}/state.yaml`

## 输入

- `.sdd/onboard.md`（项目概览 — 了解架构和规范）
- `.sdd/{需求目录}/docs/spec.md`（已通过评审）
- `.sdd/config.yaml`（项目上下文）

## 执行步骤

### 1. 分析现有架构

- 阅读项目代码结构，理解分层（controller / service / dao / model）
- 识别现有的公共组件（统一响应、全局异常处理、拦截器等）
- 识别可复用的模块

### 2. 设计实施方案

生成 `.sdd/{需求目录}/docs/plan.md`：

```markdown
# E2E 实施方案 — {功能名称}

## 1. 架构概览
- 涉及的模块/服务
- 新增模块说明
- 模块间依赖关系（文字描述或 ASCII 图）

## 2. 文件变更清单
对每个需要新增或修改的文件：
| 文件路径 | 操作 | 说明 |
|---------|------|------|
| src/.../XxxController.java | 新增 | 处理 xxx 接口 |
| src/.../XxxService.java | 新增 | xxx 业务逻辑 |
| src/.../XxxMapper.java | 新增 | 数据访问层 |
| ... | | |

## 3. 实施顺序
按依赖关系排序，先底层后上层：
1. 数据层 — Entity、Mapper、SQL
2. 业务层 — Service 接口与实现
3. 接口层 — Controller、DTO、参数校验
4. 集成 — 配置、中间件对接
5. 测试 — 单元测试、集成测试

## 4. 技术决策
对每个需要做选择的点：
### 4.1 {决策点}
- 方案 A: xxx（优：xxx / 劣：xxx）
- 方案 B: xxx（优：xxx / 劣：xxx）
- **选择**: 方案 A，原因：xxx

## 5. 风险点
| 风险 | 影响 | 缓解措施 |
|------|------|---------|

## 6. 依赖项
- 外部服务依赖
- 中间件依赖
- 需要其他团队配合的事项
```

### 3. 自检

- [ ] 文件变更清单覆盖了 spec 中所有功能
- [ ] 实施顺序合理（无前向依赖）
- [ ] 技术决策有明确理由
- [ ] 风险点已识别

## Gate 行为（对应 config 中 e2e_plan_review）

- **manual**: 展示 plan.md，等用户确认
  - 通过: 进入下一阶段
  - 不通过: 回退到 PRD_TO_SPEC
- **auto**: AI 自评 plan 可行性，不通过回退
- **flow**:
  - 方案中有多个技术选型需用户决策（非 AI 可自行决断的架构分歧）→ 暂停，列出选项，等用户选择
  - 方案完整且 AI 可自行决策 → 自动通过，继续下一阶段
- **skip**: 跳过

## 完成条件

- [x] `.sdd/{需求目录}/docs/plan.md` 已生成
- [x] 文件变更清单完整
- [x] 实施顺序明确

## 产出物

- `.sdd/{需求目录}/docs/plan.md`
