# SDD Code Review — 代码评审（阶段 7.1）

## 职责

对 TDD 实现阶段产出的所有代码进行全面评审。

## 路径约定
本阶段所有文件读写基于当前需求目录 `.sdd/{需求目录}/`：
- spec → `.sdd/{需求目录}/docs/spec.md`
- tasks → `.sdd/{需求目录}/docs/tasks.md`
- reviews → `.sdd/{需求目录}/docs/reviews/`
- state → `.sdd/{需求目录}/state.yaml`

## 输入

- TDD 阶段产出的所有代码变更（git diff）
- `.sdd/{需求目录}/docs/spec.md`（一致性校验）
- `.sdd/{需求目录}/docs/tasks.md`（完整性校验）

## 评审维度

### 1. Spec 一致性
- [ ] 每个接口实现是否与 spec 定义一致（参数、响应、错误码）
- [ ] 业务逻辑是否与 spec 描述一致
- [ ] 数据模型是否与 spec 定义一致

### 2. 代码质量
- [ ] 命名是否清晰、一致
- [ ] 方法长度是否合理（< 50 行）
- [ ] 类职责是否单一
- [ ] 是否有重复代码可提取
- [ ] 注释是否必要且准确（不写废话注释）

### 3. 安全性
- [ ] SQL 是否使用参数化查询（无拼接）
- [ ] 用户输入是否校验
- [ ] 敏感信息是否脱敏
- [ ] 权限控制是否到位

### 4. 性能
- [ ] 是否有 N+1 查询
- [ ] 是否有不必要的全表扫描
- [ ] 缓存使用是否合理
- [ ] 是否有潜在的内存泄漏

### 5. 可维护性
- [ ] 异常处理是否完善（不吞异常）
- [ ] 日志是否充足且分级合理
- [ ] 配置是否外部化（不硬编码）
- [ ] 是否易于扩展

### 6. 测试质量
- [ ] 测试是否覆盖正常流程 + 异常流程 + 边界条件
- [ ] 测试是否可独立运行（不依赖外部环境）
- [ ] mock 是否合理（不过度 mock）
- [ ] 测试命名是否清晰表达意图

## 评审输出

```markdown
# Code Review Report

## 结论: PASS / NEEDS_FIX

## 统计
- 变更文件数: N
- 新增代码行: N
- 测试覆盖率: N%

## 问题列表

### MUST FIX
1. [{文件}:{行号}] {问题描述} → {修复建议}

### SHOULD FIX
1. [{文件}:{行号}] {问题描述} → {修复建议}

### NICE TO HAVE
1. [{文件}:{行号}] {问题描述}
```

## Gate 行为

- **manual**: 展示 review report，等用户确认
- **auto**:
  - 无 MUST FIX → 通过（SHOULD FIX 生成 TODO 注释）
  - 有 MUST FIX → 回退 TDD_IMPL，只修复问题 task
- **skip**: 跳过

## 产出物

- `.sdd/{需求目录}/docs/reviews/code-review-{timestamp}.md`
