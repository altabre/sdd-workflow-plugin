# SDD PRD to Spec — PRD 转技术规格（阶段 2.1）

## 职责

将产品需求文档翻译为结构化的技术规格，作为后续所有开发工作的唯一真相源。

## 路径约定
本阶段所有文件读写基于当前需求目录 `.sdd/{需求目录}/`：
- spec → `.sdd/{需求目录}/docs/spec.md`
- inputs → `.sdd/{需求目录}/inputs/`
- state → `.sdd/{需求目录}/state.yaml`

## 输入

- `.sdd/onboard.md`（项目概览 — 先读取，了解项目架构和规范）
- PRD 文档（从 `.sdd/{需求目录}/inputs/README.md` 获取路径）
- API 设计文档 / Swagger
- DB 设计文档

## 执行步骤

### 1. 阅读并理解 PRD

- 提取所有功能点，逐条列出
- 识别隐含需求（错误处理、权限控制、并发、幂等等）
- 标记不清晰或有歧义的点（后续需人工补充）

### 2. 分析 API 设计

- 读取 Swagger/API 文档
- 列出所有需要实现的接口
- 对每个接口提取：method、path、请求参数、响应结构、错误码

### 3. 分析 DB 设计

- 读取表结构定义
- 识别实体关系
- 标注索引、约束、分表策略

### 4. 生成 spec.md

必须包含以下章节：

```markdown
# 技术规格 — {功能名称}

## 1. 概述
- 功能目标（一句话）
- 关联 PRD 章节
- 关联 JIRA ticket

## 2. 接口规格
对每个接口：
### 2.1 {接口名}
- Method: POST
- Path: /api/v1/xxx
- 请求参数:
  | 字段 | 类型 | 必填 | 说明 |
  |------|------|------|------|
- 响应结构:
  | 字段 | 类型 | 说明 |
  |------|------|------|
- 错误码:
  | code | message | 触发条件 |
  |------|---------|---------|
- 业务规则:
  1. xxx
  2. xxx

## 3. 数据模型
### 3.1 表结构
### 3.2 实体关系
### 3.3 索引设计

## 4. 业务逻辑
对每个核心流程：
### 4.1 {流程名}
- 前置条件
- 处理步骤（编号列表）
- 异常分支
- 后置动作（消息推送、日志、缓存更新等）

## 5. 非功能需求
- 性能要求（QPS、RT）
- 并发控制（锁、幂等）
- 安全约束（鉴权、脱敏）
- 监控告警

## 6. 边界条件
- 空值处理
- 极端数据量
- 依赖服务不可用时的降级

## 7. 待确认项
- [ ] 需要人工确认的问题列表
```

## 质量检查

生成后自检：
- [ ] 每个 PRD 功能点都有对应的接口或逻辑
- [ ] 每个接口都有完整的请求/响应/错误码定义
- [ ] 业务规则无歧义，可直接转为测试用例
- [ ] 待确认项已标记（不自行假设）

## 完成条件

- [x] `.sdd/{需求目录}/docs/spec.md` 已生成
- [x] 覆盖所有 PRD 功能点
- [x] 待确认项已列出

## 产出物

- `.sdd/{需求目录}/docs/spec.md`
